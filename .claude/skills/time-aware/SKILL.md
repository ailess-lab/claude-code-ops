# time-aware — 时间认知与任务排期

处理时间安排时，先建立"现在几点"的认知，再把相对时间正确换算成绝对时间。**核心：不凭印象，先 date。**

## 什么时候用

- 用户用相对时间词安排（明早 / 今晚 / 明天 / 后天 / 这周末 / 下周）。
- 用户没说具体日期时间、让你定（"到时候""合适的时候""明早"）。
- 盘点工作 / 读任务状态 / 汇报进展。
- 往 `tasks.md` 写带时间戳的任务。

## 核心行为（每次都做）

### 1. 先 `date` 拿当前精确时刻

不凭印象、不用系统 prompt 的日期（系统只给日期不给时刻）：
```bash
date "+%Y-%m-%d %H:%M %A"
```

### 2. 相对→绝对换算（基于第 1 步的时刻）

| 用户说 | 当前时刻 | 换算成 |
|---|---|---|
| 明早 / 明天早上 | **凌晨 0-6 点（睡醒前）** | **当天上午**（如 08:30，几小时后）★ 最易错 |
| 明早 / 明天早上 | 上午/下午/晚上 | 次日上午 |
| 今晚 | 任意 | 当天晚上（如 20:00）|
| 明天 | 任意 | 次日 |
| 后天 | 任意 | +2 天 |
| 这周末 | 周一-周五 | 本周六/日 |
| 下周 | 任意 | 下周一 |
| 到时候 / 合适的时候 | — | 按任务性质自行定合理时间 |

### 3. 没给具体时间 → 自行定 + 通知（不问确认）

- 按任务性质定合理时间（推送/发布类定近，调研/打磨类可远）。
- 写进 `tasks.md` 用**绝对时间戳**（`YYYY-MM-DD HH:MM`）。
- 然后**通知**用户"已定到 X 月 X 日 X 点"——通知，不是请求确认（用户已授权自行定）。

### 4. 任务时间戳一律绝对

不写"明早""1 小时后"——写 `2026-06-14 20:00`。

## 盘点工作 / 读任务状态

- **先读 `projects/<p>/state/tasks.md` + 全局任务清单**，再盘点/汇报——不凭对话记忆列。
- 汇报时核对时间戳 vs 当前时刻（哪些到期了、哪些没到、哪些过了没执行）。

## 改道表

| 问题 | 处理 |
|---|---|
| 系统 prompt 只给日期不给时刻 | 必须 `date`，别用系统日期推算 |
| 不确定"明早"指哪天 | 看当前时刻：凌晨说=当天上午，白天/晚上说=次日 |
| 相对时间有歧义 | 按当前时刻合理换算，定绝对时间，通知（不问）|
| 任务过时间戳没执行 | 核 run.sh 在跑 + 前置满足，别默认"该执行了"|
| 盘点凭记忆 | 回去读 tasks.md，别凭对话 |

## 验收

- [ ] 处理时间安排前先 `date` 拿当前时刻
- [ ] 相对时间正确换算（凌晨"明早"=当天上午）
- [ ] 没给具体时间 → 自行定绝对时间 + 通知（不问确认）
- [ ] 盘点工作先读 tasks.md，不凭记忆
