# skill-forge — 造 / 改 动作 skill

> 你是 skill 的"产房"：把一次跑通的动作，固化成专员以后能稳定复用的**操作手册**。
> 不重新发明——复杂流程可参考系统自带的官方 `skill-creator`，但**最终产物必须满足下面的项目约定**。

## 什么时候用
- 对谈主管下场跑通了一个动作（如"发某平台"），要沉淀成可复用 skill。
- 某个已有 skill 产出不对用户品味 / 覆盖不到某情况 → 优化它。

---

## 造一个新 skill

### 1. 查重（先治理，能改就不新建）
读 `.claude/skills/registry.md`。已有同类 → **改它**，不新建。确实没有 → 继续。

### 2. 定身份
- `name`：动词短语，英文小写连字符（如 `xhs-publish`）。
- `description`：写**强、具体**——「什么时候用 + 干嘛」，把触发场景写满（模型倾向漏触发，描述要"推一把"）。所有"何时用"写这里，不写正文。

### 3. 写正文（操作手册级——这是关键）
**先读 `references/示例-合格的skill.md`（骨架范例）和 `references/示例-repo-baseline.md`（完整真实领域 skill），照它们的结构和粒度写，别自由发挥。** 正文不是方法论，是**专员拿着就能照做**的手册：
- **编号步骤**，每步：做什么 / 具体命令或操作 / 校验点。精确到命令、字段、选择器，不留"你应该注意"这种虚话。
- **「常见问题与处理」改道表**：预判会出什么错、怎么改道——这是手册级和方法论级的分水岭。
- **验收清单**：交付前逐项勾的硬检查（专员的回执对照它打勾）。
- 关键节点（对外发布等）**停手等用户确认**。
- **产物落盘按约定**：skill 产出的运行内容（报告 / 数据 / 草稿）放哪、怎么命名，遵守 `CLAUDE.md` 的「运行产物放哪」，别自创位置。

### 4. 放对位置 + 自包含
- 路径 `.claude/skills/<name>/SKILL.md`（**必须，否则不被发现**）。
- 长子流程拆到 `.claude/skills/<name>/references/*.md`，**只放自己目录，不引用外部**。

### 5. 登记（治理防线 2）
`registry.md` 加一行：name / maturity（先 `草稿`）/ 一句话作用。**没登记 = 没交付。**

### 6. 接入动线
- 专员靠 `description` 自动触发——确认描述够强。
- 主管派任务时在任务里写明 skill 名——确认 name 清晰好引用。

### 7. 试跑 + 毕业
- 对谈主管**在场用真实任务试跑一次**，看产出对不对品味。不对 → 回第 3 步改。
- 连续 ≥3 次稳定过验收清单且过用户品味 → registry 里 maturity 改 `✅毕业`，专员可放心量产。

---

## 优化已有 skill
1. 读现有 skill + 问题来源（回执里的失败、用户反馈）。
2. **针对问题改**对应步骤 / 补改道表 / 调验收清单——别推倒重来。
3. registry 更新 maturity（退回 `打磨中` 或维持），记一句改了啥。

---

## 常见问题与处理

| 问题 | 处理 |
|---|---|
| 想加 `maturity`/`version` 进 frontmatter | 不行，frontmatter 只 `name`+`description`；maturity 记 registry |
| skill 想用另一个 skill 的内容 | 不跨目录引用；公共逻辑各自写，或抽成判断类 reference 各自引 |
| 描述写不强、怕漏触发 | 把"用户会怎么说、什么任务"列满，宁多写 |
| 不确定新建还是改 | 默认改（查重优先）；只有动作本质不同才新建 |
| 正文写成了方法论（"要注意 X"） | 推倒重写成"第 N 步：做 X，命令是…"——能照做才算手册 |

## 验收清单（交付前逐项勾）
- [ ] 位置 `.claude/skills/<name>/SKILL.md`
- [ ] frontmatter 只有 name + description，描述够强
- [ ] 正文是 step-by-step + 改道表 + 验收清单，不是方法论
- [ ] 自包含（references 不引用外部目录）
- [ ] registry 已登记并标 maturity
- [ ] 在场试跑过、产出对品味
