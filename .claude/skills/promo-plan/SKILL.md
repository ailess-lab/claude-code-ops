# promo-plan — 制定推广计划

> 从零到可执行的推广方案。**不做调研不动笔，不校准事实不发帖。**

## 前置：确认输入

1. 读项目 `strategy.md`（外部口径 + 核心卖点 + 平台状态 + 内容品味）
2. 读项目 `internal.md`（🔒 弱点和真实定位）
3. 读 `config/team.md`（对外口径 + 腔调 + 禁忌）
4. 读 `config/directives.md`（红线 + 阶段开关）
5. 确认项目有 `metrics/tracker.json`（基线数据）

**strategy 不存在 → 停，跑 `ops-setup` 先初始化。**

---

## 步骤

### 1. 产品事实校准（不校准不动笔）

**目的**：确保所有文案基于准确事实，不出现"73 commands"式的错误。

逐条核实 strategy 里的每个产品声称：

```
# 读仓库 README，对比 strategy 里的卖点和实际仓库内容
gh api repos/{owner}/{repo}/readme --jq '.content' | base64 -d
# 读仓库目录结构，确认声称的功能确实存在
gh api "repos/{owner}/{repo}/git/trees/main?recursive=1" --jq '.tree[] | .path'
```

对每个卖点，三连问：
1. 这个说法在仓库里有证据吗？
2. 这个数字（如 skills 数量）数过吗？
3. 这个"已支持"是真的已支持还是"计划中"？

**输出**：写一份「产品事实校准表」到 `content/事实校准.md`，格式：

```
| 原始声称 | 校准后 | 证据/来源 |
|---|---|---|
| 73 commands 精简到 17 | skills 精简（合并同类项），不是 commands | 仓库 .claude/skills/ 目录数 |
```

拿不准的标 ⚠️，**⚠️ 项不进文案**。

### 2. 竞品调研（有核心竞品时必做）

**触发条件**：strategy 里提到了竞品，或项目是 fork/基于已有项目。

**浅层**（默认）：跑 `competitive-intel` skill 的浅层模式。
**深层**（以下情况触发）：
- 项目是某个上游项目的 fork
- 竞品是同赛道头部
- 需要写对比文档给社区看

深层必读：竞品的入口文件（CLAUDE.md 等）、skill/command 目录列表、核心 skill 全文、用户 Issues 热点（评论数 top 5）。

**输出**：`intel/YYYY-MM-DD-竞品名-情报.md`（`competitive-intel` skill 产物）。

**校验点**：竞品调研做完后，列出 3 条「对定位有直接影响的发现」。如果列不出 3 条，说明调研不够深。

### 3. 平台调研（不调研不知道怎么写）

**目的**：搞清楚目标平台的社区口味、有效模式、同类产品表现。

对每个目标平台（strategy 里列的），做：

#### 3a. 同类产品表现

```
# HN：用 Algolia API 搜同类产品
curl -s "https://hn.algolia.com/api/v1/search?query={关键词}&tags=story&hitsPerPage=15&numericFilters=points>5" | python -c "import json,sys; [print(f\"pts:{h['points']} | {h['title']} | {h['objectID']}\") for h in json.load(sys.stdin)['hits']]"
```

```  
# Dev.to：搜 tag 交叉的文章
curl -s "https://dev.to/api/articles?tag={tag1}&tag={tag2}&per_page=10&top=30" | python -c "import json,sys; [print(f\"rx:{a['positive_reactions_count']} | {a['title']}\") for a in json.load(sys.stdin)]"
```

记录每个同类产品在目标平台的表现（points / reactions / comments）。

#### 3b. 社区情绪

读 2-3 个高评论同类帖子的评论区，提取：
- 正面反馈关键词
- 负面反馈关键词
- 高频质疑点
- 什么类型的内容被喷

#### 3c. 有效模式提炼

从表现最好的帖子中提取结构：
- 标题格式
- 正文结构（几段、每段干嘛）
- 开头怎么起、收尾怎么收
- 有没有首条评论、评论内容

**输出**：`intel/YYYY-MM-DD-平台冷启动调研.md`

**校验点**：每个目标平台至少看了 3 个同类案例。如果 API 限流或工具不可用，标注"调研受阻"并列出缺失信息——**不跳过、不猜**。

### 4. 叙事定位 + 多方向发散（核心步骤，基于 1+2+3 的发现）

**这不是写一个定位，是发散出多个方向。**

#### 4a. 纪律框架（先定边界）

**必填项**：
| 项 | 内容 |
|---|---|
| 给谁 | 目标用户画像 |
| 解决什么 | 具体痛点（不是"帮助开发者"，是"解决 Codex 长会话减速"） |
| 怎么解决 | 核心功能/创新点 |
| 为什么不选竞品 | 具体差异 |
| 核心卖点排序 | ①②③ 按强度排 |

**叙事纪律**（六条，每条必须写）：
1. 绝对不说的叙事
2. 绝对不提的数字/声称
3. 对竞品的态度
4. 语气和腔调（参照 team.md）
5. 主动暴露的局限（至少 2 条）
6. 预期管理（现实天花板，不画饼）

**校验点**：每条卖点都能回答"证据在哪？"——答不出 = 删。

#### 4b. 方向发散（至少 5 个，目标 8-10 个）

基于调研发现，列出所有可行的叙事方向。每个方向一句话：

| # | 方向名 | 一句话 | 假设 | 目标平台 |
|---|---|---|---|---|
| 1 | ... | ... | 为什么这个方向有受众（引用调研数据） | ... |

**方向怎么找**：
- 从调研发现的痛点出发（如"permission fatigue"已被命名）
- 从平台高匹配帖出发（如 r/SideProject "I got tired of X so I built Y"）
- 从竞品缺口出发（如 CCGS #24 求 Codex 支持）
- 从技术卖点出发（如多窗口 Lane 架构创新）
- 从社区情绪出发（如 HN 反过度工程 869 pts）
- 从用户自己的表达出发（用户怎么跟朋友介绍这个项目的）

**不停追问**：这个方向还能继续吗？还有别的角度吗？还有别的参考吗？发散够了再收敛。

#### 4c. 方向筛选

对每个方向，三维度评分：

| 维度 | 评分依据 |
|---|---|
| **情绪共鸣** | 调研数据里有没有对应的受众证据（帖子分数/评论数） |
| **平台匹配** | 目标平台有没有成功案例（同类帖子的分数和格式） |
| **风险** | 踩不踩红线、会不会引战、事实够不够硬 |

筛选后保留 3-5 个强方向，淘汰弱的。

#### 4d. 样稿撰写

每个强方向写一篇样稿（标题 + 正文 2-3 段），不需要完整，够评价即可。

#### 4e. 评价优选

**如果有 `content-review` skill**：用它评价所有样稿，输出评分和推荐。
**如果没有**：自己做初筛，给用户看 3-5 个样稿 + 评分 + 推荐理由，让用户拍板。

**⚠️ 不自己定最终方向——用户拍板。但自己要做完分析、写完样稿、给好推荐再交给用户。**

### 5. 各平台文案撰写（基于 4e 优选的方向）

**格式**：每个平台一个文件，放入 `content/{平台}/`。

**每个平台选定 1 个方向**（从 4e 优选结果中选），写完整稿件。

**每篇文案必须包含**：
- 标题（2-3 个候选）
- 正文（贴合平台口味 + 选定方向）
- 预期社区问题 + 回答（至少 3 个）

**各平台格式参考**（基于平台调研）：

| 平台 | 有效格式 | 素材要求 | 无效格式 |
|---|---|---|---|
| HN Show HN | "Hey HN!" + 技术深度 + 诚实局限 + 首条评论 | 文字 + 截图 | 只贴链接、不写正文 |
| Reddit r/godot | 工具分享（"I made a tool"） | **图片就够**（3k pts 案例验证） | 硬广、产品发布帖 |
| Reddit r/SideProject | "I was tired of X so I built Y" | 文字 + 图片即可（4,597 pts 纯文本验证） | 硬广 |
| Dev.to | 教程/学习记录（"How I..."） | 文字 + 代码块 | 产品发布帖（几乎无人看） |

> 详细素材需求见 `intel/` 下的平台调研报告「视频地图」段落。

**校验点**：
- 文案里的每个产品声称都在「产品事实校准表」里有对应行
- 没有校准表里标 ⚠️ 的内容
- 没有违反叙事纪律的措辞
- 对竞品忠实用户的应对写进了互动计划

### 6. QA 准备（不准备就被问住）

**输出**：`content/QA.md`

分类覆盖：
1. **身份/动机类**（你是谁、为什么做这个）
2. **产品事实类**（最容易出问题——每个声称都要有回答）
3. **对比/竞品类**（和 XXX 比怎么样）
4. **质疑/负面类**（"这只是 fork""这是 AI slop"）
5. **技术深入类**（HN 高频——实现细节）
6. **回答纪律**（绝对不说什么、必须做到什么）

**每个回答格式**：问题 → 回答 → ⚠️ 不说（红线）→ 证据来源

**校验点**：至少覆盖 15 个预期问题。如果有核心竞品，至少 3 个问题是关于竞品忠实用户可能的质疑。

### 7. 风险预案 + 互动计划

**风险预案**：列出所有可预见的风险 + 具体应对 + 依据来源（来自哪次调研的哪个发现）。

**互动计划**：
| 平台 | 首条回复时限 | 互动要点 | 特殊应对 |
|---|---|---|---|
| HN | 30 分钟 | 技术深度 | ... |
| Reddit | 24 小时 | 实用性 | ... |

**特殊应对场景**（如果有核心竞品）：
- 竞品忠实用户来辩护
- 被说"只是 fork / 改动太少"
- 竞品社区的反感

### 8. 发布排期 + 素材清单

**排期**：
| 天数 | 平台 | 时间窗口 | 角度 |
|---|---|---|---|

**素材清单**：每个素材标注状态（✅ / ⬜）和负责人。

**仓库准备清单**（发布前必须全 ✅）：
- README 数据校准
- Quick Start 实测可跑
- Release tag
- Org 信息

**预期管理**：写明基于同类产品数据的现实天花板。不画饼。

### 9. 写推广方案文件

把步骤 4-8 合成一个完整文件：`content/PROMO-PLAN-{标识}.md`

包含：
1. 发布前准备清单（仓库 + 社区 + 素材）
2. 叙事定位（卖点 + 纪律）
3. 发布时间策略
4. 社区互动计划
5. 跟进节奏
6. 后续持续推广
7. 风险预案
8. 素材清单汇总

**校验点**：推广方案里每个结论都标注了"基于什么调研发现"。

### 10. 发布后追踪与倒查（发布 ≠ 完事）

> 6/13 教训：发布后 42h 零互动，正确反应是倒查"链接死了吗/平台对吗/账号够吗/文案对吗"，不是记一笔"渠道不对"就过。**记录 ≠ 尽职。**

发布后按排期收数据（T+8h / T+24h / T+48h 三个快照）：每平台看互动数（reactions/upvotes/comments）、流量、是否被限流/折叠。

**零互动或极低（远低于「预期管理」写的天花板）→ 强制倒查，四项不放过：**

| 查什么 | 怎么查 | 常见死因 |
|---|---|---|
| 链接活吗 | 无痕窗口打开仓库链接 | 仓库私有、链接拼错、平台吞链接 |
| 平台对吗 | 这平台同类内容平时多少互动 | 平台口味不符（如 Dev.to 要教程，pitch 类零互动） |
| 账号够吗 | 账号 karma / 历史 / 初始化状态 | 新号没养号被 spam filter 拦，发帖即折叠 |
| 文案对吗 | 标题/格式/角度对照平台有效模式 | 标题像广告、格式踩雷 |

**倒查产出**：每条结论写进回执，**会改决策的写进 insights**（"Dev.to 要教程不要 pitch"是 insight，"今天 0 反应"是流水）。**只记一个 0 然后过 = 违反这一步。**

定位死因 → 改对应环节（修链接/换平台/补养号/改文案）重发或放弃，**不是原样再发一遍**。

## 常见问题与处理

| 问题 | 处理 |
|---|---|
| 搜索/Web API 限流 | 标注"调研受阻"并列缺失信息，不跳过不猜。等限流恢复后补。 |
| 产品事实校准发现有假声称 | 改声称，不删改行。假声称不进文案。 |
| 没有核心竞品 | 跳过深层调研，但浅层同类工具扫描仍要做 |
| 用户急着发、不想等调研 | 调研是防弹衣不是装饰。告诉用户：跳过调研 = 文案踩雷风险自担。 |
| 调研做完发现竞品忠实用户很强势 | 写进风险预案，准备专门应对话术。不回避。 |
| 平台上同类产品全是低互动 | 这就是现实天花板。写进预期管理，不画饼。 |
| 文案写完后产品又改了 | 回到步骤 1 重新校准受影响的声称，更新文案。 |
| 多个推广方案（如 A/B 方案） | 每个方案单独一个文件，共享同一套调研和定位。 |
| 推广方案写完了但发布条件不满足 | 方案写完 ≠ 能发。标"待发布条件满足"。方案本身可以交付。 |
| **调研数据太少（如 Reddit 只抓到 3 帖）** | 换方式（old.reddit.com 比新版更容易抓取；用搜索而非热门排序）。标明样本量。 |
| **写出的文案用户觉得不如自己写** | 正常——创作者偏袒自己的文字。解决方式：① 采集用户自己的写作样本作为腔调校准 ② 用 `content-review` skill 做第三方评价 ③ 不替代用户写，而是提供多个方向供用户改 |
| **过早收敛成一个方向** | 回到 4b 继续发散。运营是试错不是证明题。至少 5 个方向再筛。 |
| **把分析工作丢给用户** | 不对。自己做完分析、写完样稿、给好推荐再交给用户拍板。用户只做选择题不做填空题。 |

## 验收清单（回执逐项勾）

- [ ] 产品事实校准表写了，每个声称有证据
- [ ] 竞品调研做了（有竞品时），3 条定位级发现列出了
- [ ] 平台调研做了，每个目标平台 ≥3 个同类案例，含高匹配度帖子
- [ ] 素材地图写了（各平台最低素材要求 + 最优素材 + 制作指南）
- [ ] 叙事纪律 6 项全填了
- [ ] **方向发散 ≥5 个，每个有假设和平台匹配度**
- [ ] **方向筛选做了，3 维度评分（情绪共鸣 / 平台匹配 / 风险）**
- [ ] **样稿写了（每个强方向至少一篇），不是只有方向名**
- [ ] 各平台完整文案写了（基于优选方向），每个声称可追溯到校准表
- [ ] QA 文档 ≥15 个问题，竞品忠实用户场景 ≥3 个
- [ ] 风险预案有依据来源
- [ ] 发布排期 + 素材清单 + 仓库准备清单写了
- [ ] 预期管理写了，基于同类产品实际数据
- [ ] 所有产物放对位置（content/ 和 intel/）
- [ ] 推广方案文件已写入 `content/PROMO-PLAN-{标识}.md`

### 如已发布（发布后追踪，逐项勾）

- [ ] T+8h / 24h / 48h 数据快照收了
- [ ] 零/极低互动已倒查四项（链接/平台/账号/文案），不只记一个 0
- [ ] 倒查结论里会改决策的写进了 insights
